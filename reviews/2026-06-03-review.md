This layer is a thin, well-structured proxy in front of the broker. Authorization is
broker-authoritative; the in-process write gates that exist (`orgs.rs`) are applied
correctly, and there is no fail-open path here. No Critical or Security findings.

The two Major findings are both robustness issues, not security ones:

- **M1** — a best-effort local cache write (`cache_installation_access`) is propagated
  with `?`, so a local disk error makes the entire installation listing/inspection
  fail even though the network fetch succeeded.
- **M2** — `invite_user_to_organization_for_installation` is the only command in the
  module that is not wrapped in `spawn_blocking`; it performs two blocking broker
  round-trips directly on the IPC thread (Standard V violation).

## Findings

---

### M1 — `github/orgs.rs:32-34`, `github/orgs.rs:54`

**A best-effort cache write aborts the user-facing installation fetch**

```rust
let installations: Vec<GithubAppInstallationInfo> =
    broker_get_json_with_session(&client, "/api/github-app/installations", &session_token)?;
for installation in &installations {
    cache_installation_access(&app, installation)?;   // ← local write error fails the whole command
}
Ok(installations)
```

`list_accessible_github_app_installations` (and `inspect_github_app_installation` at
`orgs.rs:54`) fetch installation data from the broker — the user-critical part — and
then write a local access-snapshot cache. The cache write is propagated with `?`, so a
local failure (read-only disk, permissions, full volume) makes the entire command
return `Err` and the user sees no installations at all, despite a successful fetch.

The cache is an optimization (`installation_access.rs` re-fetches on a 60s TTL anyway).
A failure to persist it should not block the user from listing or inspecting their
installations.

**Impact**: A transient or persistent local-write problem degrades a non-critical
optimization into a hard failure of the app's entry-point screen (installation list).

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Treat caching as best-effort: log the error and continue rather than `?`. e.g. `if let Err(error) = cache_installation_access(&app, installation) { /* log */ }` | The fetch is authoritative; the cache is rebuilt on next access. The user-facing read should not fail on a local-write error. |
| B | Collect cache errors and return the installations plus a non-fatal warning event | Surfaces the problem without blocking; more plumbing |

**Recommended**: A

### M2 — `github/orgs.rs:103-127`

**`invite_user_to_organization_for_installation` blocks the IPC thread**

```rust
#[tauri::command]
pub(crate) fn invite_user_to_organization_for_installation(
    /* ... */
) -> Result<GithubOrganizationInvitation, String> {
    ensure_installation_allows_member_management(&app, installation_id)?;  // blocking broker round-trip
    let client = github_client()?;
    broker_post_json_with_session(/* ... */)                              // blocking broker round-trip
}
```

Every other command in `orgs.rs` and `repos.rs` is `async` and wraps its blocking work
in `tauri::async_runtime::spawn_blocking`. This one is a plain synchronous
`#[tauri::command]` that performs two blocking broker calls (`ensure_*`, which itself
does a broker round-trip via `refresh_installation_access_snapshot`, and the POST)
directly on the IPC worker. This violates the constitution's **Standard V (IPC
Non-Blocking)** and the AGENTS.md rule against blocking the IPC call path on network
operations.

**Impact**: A slow or unresponsive broker during an invite blocks the command thread
(up to the 30s client timeout), freezing the calling path with no progress signal —
the exact failure mode `spawn_blocking` exists to prevent everywhere else here.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Convert to `async` and wrap the body in `spawn_blocking`, matching `add_organization_admin_for_installation` and its siblings | Restores consistency; takes the blocking work off the IPC thread |

**Recommended**: A

### m1 — `github/repos.rs:39-83`, `github/orgs.rs:62-79`

**List endpoints fail entirely if a single item fails to deserialize**

`list_gnosis_projects_for_installation`, `list_gnosis_glossaries_for_installation`,
`list_gnosis_qa_lists_for_installation`, and `list_organization_members_for_installation`
deserialize the broker response directly into `Vec<T>`. `serde` fails the **entire**
array if any one element is missing a required field or has an unexpected shape, so one
malformed repo/member discards the whole list.

This is in tension with the AGENTS.md guidance: _"GitHub installation repository list
responses are paginated and may omit repos in degraded states — tolerate missing repos
gracefully (`e1312a69`)."_ The response tolerance focus of this batch makes it worth
recording.

Likelihood is **moderate-to-low** because the broker normalizes these payloads (the
client is not parsing raw GitHub API output), so the contract is tighter than a direct
GitHub call would be.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Deserialize into `Vec<serde_json::Value>` (or `Vec<IgnoredAny>`-guarded items) and skip/log elements that fail to convert to the typed struct | One degraded item no longer hides the rest of the list |
| B | Leave as-is and rely on the broker to guarantee well-formed arrays; add a comment recording that assumption | Lower effort; depends entirely on broker contract discipline |

**Recommended**: A for the resource lists (`repos.rs`); B is acceptable for members if the broker contract is considered authoritative.

### m2 — `github/orgs.rs:118`, `:184`, `:234`, `:261`, `:286`, `:399` (and siblings)

**Path parameters interpolated into broker URLs without percent-encoding**

`org_login` and `username` are interpolated straight into the request path via
`format!`, e.g.:

```rust
&format!("/api/github-app/installations/{installation_id}/orgs/{org_login}/members/{username}")
```

By contrast, `search_github_users_for_installation` (`orgs.rs:89-90`) correctly
percent-encodes its query parameter with `url::form_urlencoded::byte_serialize`. The
path parameters get no such treatment. `broker_path_url` splits the incoming path on
`?` before assigning the query, so a value containing `?`, `/`, or `#` would alter the
request path or query rather than being treated as a literal segment.

In practice these values are GitHub logins (constrained to `[A-Za-z0-9-]`), so the
exposure is small — this is a **defense-in-depth and consistency** finding, not an
active injection vector.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Percent-encode path segments (e.g. a small `encode_path_segment` helper using `percent-encoding`/`form_urlencoded`) before interpolation, mirroring the search query handling | Consistent encoding discipline; removes the latent path/query-manipulation surface |
| B | Validate `org_login`/`username` against the GitHub login charset and reject otherwise | Explicit, but adds validation rather than reusing standard encoding |

**Recommended**: A

### m3 — `github/app_auth.rs:36` vs `broker.rs:12`

**Two near-identical HTTP client factories are both used for broker calls**

There are two blocking-client factories:

- `broker_client()` (`broker.rs:12`) — 30s timeout, no user agent. Used by `broker_auth.rs`.
- `github_client()` (`github/app_auth.rs:36`) — 30s timeout, `user_agent("GnosisTMS")`. Used by **all** `orgs.rs` / `repos.rs` commands.

Both factories build clients that talk to the **broker**, but the module split uses them
inconsistently: `broker_auth.rs` uses `broker_client()`, while the GitHub-named commands
use `github_client()` for their broker requests. This is the same kind of duplication
flagged as M3 in Batch 1 (`random_token`), and it means timeout/header policy for broker
traffic now lives in two places.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Pick one factory for broker traffic (prefer `broker_client()`, since all calls here are broker calls) and route all broker requests through it; keep `github_client()` only if/when there is a direct-to-GitHub call that needs the GnosisTMS user agent | Single timeout/header policy point for broker traffic |
| B | Add a comment documenting why two factories exist and which to use | Lower effort; preserves the split |

**Recommended**: A (low priority — both have timeouts, so this is hygiene, not a hang risk)

### m4 — `github/repos.rs:85-245`

**Resource-mutation commands have no in-process write gate, unlike `orgs.rs` mutations**

`create_gnosis_project_repo`, `create_gnosis_glossary_repo`, `create_gnosis_qa_list_repo`,
`rename_gnosis_project_repo`, `mark_gnosis_project_repo_deleted`, `restore_gnosis_project_repo`,
and the three `rollback_created_*` commands call the broker directly with **no**
`ensure_installation_allows_*` check, whereas every mutating command in `orgs.rs` calls
`ensure_installation_allows_member_management` / `_team_management` first.

This is most likely **intentional**: the constitution's **Standard II** places
resource-management gating in `team_metadata_local.rs` command bodies (which front these
repo operations), and the broker independently enforces org-admin authorization for
mutating actions. So the gate is expected to live one layer up, not here.

Because the gating path is in `team_metadata_local.rs` (Batch 8), this is recorded as a
**verification item**, not a confirmed gap.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Confirm during Batch 8 that every caller of these `repos.rs` commands passes through an `ensure_installation_allows_*` gate in `team_metadata_local.rs`; add a one-line comment on the `repos.rs` commands pointing to where the gate lives | Documents the layering so a future reader does not mistake the absence for a gap (and re-flag it) |
| B | Add a redundant `ensure_installation_allows_*` gate directly in `repos.rs` | Defense-in-depth, but duplicates the gate and adds a broker round-trip the upper layer already performs |

**Recommended**: A

## What Was Done Well

- **Consistent `spawn_blocking` usage** — every command except the invite (M2) takes
  blocking broker/network work off the IPC thread, with the join error wrapped in an
  actionable message.
- **Actionable, specific error messages** — every command wraps task-join failures with
  a domain-specific string (`"Could not run the project listing task: …"`), consistent
  with the AGENTS.md error-handling convention. No raw panic strings.
- **Correct in-process write gates** — mutating `orgs.rs` commands call the right
  `ensure_installation_allows_*` gate (member vs. team management) before acting;
  `leave_organization_for_installation` correctly has no gate (self-service).
- **Search query is percent-encoded** (`orgs.rs:89-90`) — the one user-supplied
  free-text parameter is handled correctly (the path-param gap in m2 is the inconsistency).
- **Good Option discipline in `types.rs`** — optional GitHub fields are `Option<_>`, and
  count fields use `#[serde(default)]`. No `deny_unknown_fields`, so the structs tolerate
  forward-compatible additions from the broker.
- **`github_client()` and `broker_client()` both set a 30s timeout** — the Batch 1 M2
  hang risk does not regress in this layer (the m3 finding is duplication, not a missing
  timeout).
