Both Security findings are in `telemetry-scrub.js` — the last-line-of-defense scrubber
before data leaves the process — and both have been fixed as part of this review.

- **S1** — the breadcrumb scrubbing block in `scrubEvent` checks `Array.isArray(event.breadcrumbs)`,
  but the Sentry SDK event format uses `{ values: [...] }`, not a bare array. In real Sentry events,
  breadcrumbs pass through unscrabbed. A belt-and-suspenders `beforeBreadcrumb: () => null` in
  `telemetry.js` prevents breadcrumbs from being added in the first place, but the scrubber's last-line
  defense was silently broken. **Fixed.**

- **S2** — fine-grained GitHub PATs (`github_pat_…`) are not covered by the existing `gh[pousr]_`
  pattern and would pass through `scrubString` unredacted if present in an error message. **Fixed.**

The Major finding is a trust-boundary edge case in `auth-flow.js`: all errors from the
broker session verification call fall through to an "offline success" state, including
explicit authentication rejections. This is tolerable under the app's offline-first design
(the broker enforces auth on every command), but it is worth tracking.

No consent gate bypass paths found. No main-thread blocking found. No crypto correctness
issues found.

### S1 — `telemetry-scrub.js:173`

**Breadcrumb scrubbing is dead code — Sentry events use `{ values: [] }`, not a bare array**

```js
// before (broken)
if (Array.isArray(event.breadcrumbs)) {
  event.breadcrumbs = event.breadcrumbs.map(…);
}
```

`@sentry/browser`'s event type is `{ breadcrumbs?: { values?: Breadcrumb[] } }`. The
`Array.isArray` check is permanently false in real Sentry events, so breadcrumb messages
and data were never scrubbed by `scrubEvent`. A path like `/Users/alice/project` or a
token in a breadcrumb message would leave the process unredacted.

**Mitigating control**: `initTelemetry` passes `beforeBreadcrumb: () => null` to
`sentry.init`, which prevents any breadcrumb from being added. Breadcrumbs on real events
were therefore empty, and the scrubbing gap did not cause an active leak. However, the
scrubber is the last line of defence; a configuration change (e.g. re-enabling
breadcrumbs) would silently bypass it.

**Fix applied**: updated `scrubEvent` to handle both the legacy bare-array format (used in
tests and the pre-4.x Sentry SDK) and the canonical `{ values: [] }` object format. The
`values` array is scrubbed and written back into the correct structure.

```js
// after
const rawBreadcrumbs = Array.isArray(event.breadcrumbs)
  ? event.breadcrumbs
  : Array.isArray(event.breadcrumbs?.values)
  ? event.breadcrumbs.values
  : null;
if (rawBreadcrumbs !== null) {
  const scrubbed = rawBreadcrumbs.map(…);
  if (Array.isArray(event.breadcrumbs)) {
    event.breadcrumbs = scrubbed;
  } else {
    event.breadcrumbs = { ...event.breadcrumbs, values: scrubbed };
  }
}
```

**Status**: ✅ Fixed in this review session.

### S2 — `telemetry-scrub.js:36`

**Fine-grained GitHub PATs (`github_pat_`) are not covered by SECRET_VALUE_PATTERNS**

```js
// before
/\bgh[pousr]_[A-Za-z0-9]{16,}\b/g, // GitHub tokens
```

The existing regex covers classic GitHub token prefixes (`ghp_`, `gho_`, `ghu_`, `ghs_`,
`ghr_`). Fine-grained personal access tokens introduced in 2022 use the prefix
`github_pat_` (e.g. `github_pat_11ABCDE…`) and are not matched. If a fine-grained PAT
appeared in an error message or stack trace, it would be transmitted unredacted to Sentry.

This is relevant because the GitHub App permissions feature in this batch covers
repository/org administration — contexts where fine-grained PATs are increasingly common
as an alternative to classic tokens.

**Fix applied**: added the pattern as the first entry in `SECRET_VALUE_PATTERNS`:

```js
/\bgithub_pat_[A-Za-z0-9_]{16,}\b/g, // GitHub fine-grained PATs
```

**Status**: ✅ Fixed in this review session.

### M1 — `auth-flow.js:156-170`

**All `inspect_broker_auth_session` errors fall through to an offline-success state**

```js
try {
  const profile = await invoke("inspect_broker_auth_session", { sessionToken: … });
  // happy path — sets status: "success" with verified session
} catch (error) {
  // ← error is completely ignored
  state.auth = {
    status: "success",
    message: `Signed in as @${session.login}.`,
    session,           // unverified stored session
    …
  };
  …
  void loadUserTeams(render);  // will fail at broker
}
```

The catch block is intended as an offline fallback: if the Tauri backend is unreachable,
the app should continue with the cached session. That design is sound. However, the same
path is taken for *any* error from `invoke`, including an explicit 401/403 from the broker
(revoked token, expired session, account suspended). In those cases the user is shown
"Signed in as @login" and navigated to the teams screen, where every subsequent command
will silently fail with auth errors until the user notices.

`classifySyncError` / `handleSyncFailure` (imported at the top of the file) provide the
right primitives to distinguish network failures from auth rejections, but they are not
called here.

**Impact**: An explicitly revoked or expired session is indistinguishable from a temporary
connectivity loss at restore time. The user may work for an indefinite time in an
effectively "broken" authenticated state before the degradation surfaces.

**Note on trust boundary**: this is not a security bypass — the broker enforces auth
independently on every command. No data can be read or written through a rejected token.
The impact is purely UX correctness and failure transparency.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Call `classifySyncError(error)` inside the catch; if the result is an auth-rejection class, set `status: "error"` and route to the sign-in screen rather than the offline success path. Network/timeout errors still fall through to the offline path. | Uses the existing error-classification machinery; clearly separates "offline" from "rejected". |
| B | Log the error and leave the offline path as-is; add a UI signal (e.g. banner) when commands fail with auth errors after restore | Less invasive; the broker error path already surfaces failures at command time |

**Recommended**: A

**Status**: Open.

### m1 — `team-ai-crypto.js:62`

**`pemBody` splits on `\n` only; Windows-style PEM data (`\r\n`) would fail at WebCrypto import**

```js
function pemBody(value) {
  return String(value ?? "")
    .trim()
    .split("\n")
    .filter((line) => !line.startsWith("-----"))
    .join("");
}
```

PEM content with Windows-style line endings (`\r\n`) would leave `\r` characters at the
end of each line after `split("\n")`. These `\r` characters are valid base64 content
characters but do not represent base64 data, so `base64Decode` would silently produce
incorrect bytes and `importBrokerPublicKey` / `importMemberPrivateKey` would fail at the
WebCrypto import step with a `DataError`.

**Actual risk**: low. Keys are generated locally by `generateTeamAiMemberKeypair` and
roundtripped through `bytesToPem`, which uses `\n` throughout. A `\r\n`-formatted PEM
would have to be introduced by an external tool or manual copy-paste. The failure mode is
a hard import error rather than a silent decryption failure, so it surfaces quickly.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Replace `split("\n")` with `split(/\r?\n/)` | Single-character change; tolerates both line-ending conventions and matches standard PEM parsing practice |

**Recommended**: A (low priority; fix alongside any future `team-ai-crypto.js` work)

**Status**: Open.

### m2 — `telemetry.js:208`, `telemetry.js:226`

**`command`, `operation`, and `reason` tag values bypass pre-scrubbing; rely on `scrubEvent`**

```js
sentry.captureMessage(message, {
  level: "error",
  tags: { source: "command-failure", command: String(command ?? "unknown") },
});
```

The `command`, `operation`, and `reason` values are set as Sentry tags without being run
through `scrubString` first. They are scrubbed by `scrubEvent` → `scrubData` before
transmission (tags are handled at line 195-197 of `telemetry-scrub.js`), but the scrubbing
is by key-name pattern (`isSensitiveKey`) and string value patterns — not by semantic
intent. If a caller passes a non-stable string as `command` (e.g. a path or message
fragment), the key name `"command"` does not trigger key-name redaction, and the value
would only be caught by the regex patterns in `scrubString`.

**Actual risk**: low. All current callers pass stable constant strings from the codebase
(`"invoke"`, named command strings). No dynamic content is passed today.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Wrap tag values through `scrubString` before assigning them to the event | Belt-and-suspenders; makes the contract explicit and removes the reliance on `scrubEvent` for values that are meant to be stable identifiers |

**Recommended**: A (low priority)

**Status**: Open.

## What Was Done Well

- **`auth-storage.js`**: minimal credential surface; validates `sessionToken` and `login`
  before accepting stored data; catch blocks are documented with intent comments.
- **`github-app-permissions.js`**: clean, pure logic for permission-level checking; the
  multi-key `reduce` approach for aliased permissions (e.g. `custom_properties` /
  `repository_custom_properties`) is correct and well-contained.
- **`team-ai-crypto.js`**: RSA-OAEP 2048/SHA-256 is the right primitive for key
  transport. The algorithm version tag (`rsa-oaep-sha256-v1`) enables forward-compatible
  key rotation. The chunked base64 encoder avoids stack overflow on large payloads. Strict
  empty-check on decrypted output catches silent decryption failures.
- **`telemetry.js`**: `safe()` wrapper throughout; telemetry can never block or throw from
  a command handler. Dynamic SDK import keeps startup cost zero until consent.
  `defaultIntegrations: false` + `beforeBreadcrumb: () => null` eliminates the largest
  auto-data-collection surfaces. `reportCommandFailure` and `reportBackendNonfatalError`
  both gate on `gateOpen()` — no routine events sent before disclosure is shown.
- **`telemetry-consent.js`**: clean two-factor gate (`isDisclosureShown() && isTelemetryEnabled()`).
  Opt-out model is clearly documented. Dependency-injection store pattern makes the module
  unit-testable without mocking globals.
- **`telemetry-disclosure-flow.js`**: no consent gate bypass paths found. `markDisclosureShown`
  is called before `initTelemetry`, ensuring the gate is set before any send attempt.
  `updateTelemetryDisclosureEnabled` correctly only updates in-memory state; the actual
  write is deferred to `saveTelemetryDisclosureSettings`.
