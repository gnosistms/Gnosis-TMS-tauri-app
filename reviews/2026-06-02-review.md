### C1 — `installation_access.rs:271–276`

**ensure_repo_allows_writes is fail-open when the repo path has no installation-id component**

```rust
pub(crate) fn ensure_repo_allows_writes(app: &AppHandle, repo_path: &Path) -> Result<(), String> {
    if let Some(installation_id) = installation_id_from_path(repo_path) {
        ensure_installation_allows_chapter_writes(app, installation_id)?;
    }
    Ok(())   // ← silently passes when no installation-<id> segment found
}
```

`installation_id_from_path` scans path components for a segment matching `installation-<integer>`. If no such component is found — due to a differently-structured path, a symlink, or any future code path that passes a non-standard path — the function returns `Ok(())` without checking GitHub installation write access. The sole caller is `git_commit_as_signed_in_user_with_metadata` (`git_commit.rs:70`). Every git commit goes through this gate.

AGENTS.md states: _"Never assume a logged-in user has write access. Even owners can have a GitHub App installation in a degraded permission state. Check explicitly."_ The function name `ensure_*` establishes a guarantee; the fail-open is a violation of that contract.

**Impact**: Any user who can trigger a commit via a repo path that does not structurally contain an `installation-<id>` directory segment bypasses the GitHub installation write-access check entirely.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Return `Err` when `installation_id_from_path` returns `None`: `let installation_id = installation_id_from_path(repo_path).ok_or_else(\|\| "Could not determine the GitHub installation for this repository.".to_string())?;` | Enforces fail-closed. Any path not under a known installation directory is blocked. |
| B | Add a debug assertion that panics in dev builds and returns an error in release builds | Less disruptive if some legitimate paths truly lack an installation ID, but requires confirming no such paths exist |
| C | Rename to `try_ensure_repo_allows_writes`; add a hard `enforce_*` variant that errors on `None`, making the fail-open explicit at call sites | Forces callers to opt in to the bypass |

**Recommended**: A

### S1 — `ai_secret_storage.rs:124–135`

**Stronghold encryption key is fully deterministic and derived from public inputs; KDF work factor is zero**

```rust
fn stronghold_password(snapshot_path: &Path) -> Vec<u8> {
    let mut hasher = Sha256::new();
    hasher.update(b"gnosis-tms-ai-provider-secrets");
    hasher.update(snapshot_path.to_string_lossy().as_bytes());
    hasher.finalize().to_vec()
}

fn open_stronghold(snapshot_path: &Path) -> Result<Stronghold, String> {
    try_set_encrypt_work_factor(0).map_err(...)?;   // ← disables all key-stretching
    Stronghold::new(snapshot_path, stronghold_password(snapshot_path))
    ...
}
```

Two compounding problems: (1) The key is `SHA-256(hardcoded_string || file_path)`. Both inputs are fully public — the constant ships in the binary, the path is predictable from OS app-data conventions. Any process that can read the `.hold` file can independently compute the key. (2) `try_set_encrypt_work_factor(0)` explicitly disables Argon2 key-stretching, removing the computational barrier even for brute-force scenarios.

**Impact**: The `.hold` file containing all AI provider API keys (OpenAI, Anthropic, Gemini, DeepSeek) is encrypted in name only. Any same-user local process can decrypt it without brute-force.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Generate 32 random bytes via `OsRng` on first open; store them in the OS credential store (`keyring` crate — keychain on macOS, Secret Service on Linux, Credential Manager on Windows); load on subsequent opens. Restore default work factor. | Provides a genuine secret unknown to other processes. OS keychains are the correct primitive for this. |
| B | Mix in a randomly-generated secret stored in a `chmod 0600` file alongside the `.hold` | Better than current but still vulnerable to same-user reads |
| C | Keep key derivation; restore default work factor (remove `try_set_encrypt_work_factor(0)`) | Addresses the secondary issue only; root problem (deterministic key) remains |

**Recommended**: A

### S2 — `broker_auth_storage.rs:37–41`

**Broker session bearer token written as plaintext JSON with no file-mode restriction**

```rust
let contents = serde_json::to_string(&session)
    .map_err(|error| format!("Could not encode the broker session: {error}"))?;
fs::write(&session_path, contents)   // ← creates with process umask, no 0600
    .map_err(|error| format!("Could not save the broker session: {error}"))?;
```

`BrokerSession` includes `session_token` (a GitHub App bearer token). It is written verbatim to `broker-auth-session.json` with the process umask (typically `0644` on Linux). Any same-user process can read it and impersonate the user against the broker, including fetching GitHub installation access tokens for all organizations the user has access to.

Additionally, the token's expiry is not validated on load (`broker_auth_storage.rs:18–19`); a stale token from disk is returned to callers without a freshness check.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Store `session_token` in OS credential store (`keyring`); store only display fields (`login`, `name`, `avatar_url`) in JSON for UI rendering | Removes token from filesystem entirely. Standard practice for desktop OAuth tokens. |
| B | Store token in existing Stronghold (inherits S1 weakness until S1 is fixed) | Reuses infrastructure; weaker than A |
| C | Set file permissions to `0600` post-write on Unix (`std::os::unix::fs::PermissionsExt`) | Defense-in-depth; does not protect against same-user reads (primary threat) |

**Recommended**: A

### S3 — `installation_access.rs:253–269`

**ensure_snapshot_allows_team_ai_access uses a denylist; unrecognized role strings pass**

```rust
fn ensure_snapshot_allows_team_ai_access(snapshot: &InstallationAccessSnapshot) -> Result<(), String> {
    if is_read_only_membership_role(snapshot.membership_role.as_deref()) {
        return Err(READ_ONLY_TEAM_AI_ERROR.to_string());
    }
    if snapshot.membership_role.as_deref().unwrap_or_default().trim().is_empty() {
        return Err(UNVERIFIED_TEAM_ACCESS_ERROR.to_string());
    }
    Ok(())   // ← "unknown" role (normalized by normalized_membership_role) passes here
}
```

`normalized_membership_role` maps unrecognized role strings to `Some("unknown")`. A role of `"unknown"` is non-empty and not a viewer alias, so it passes this gate. All other content-write and management checks in this file use explicit allowlists (e.g. `snapshot_can_write_content` requires `"translator" | "admin" | "owner"` explicitly). This function is the only exception. A future server-side role addition, misconfiguration, or tampered cached value silently widens AI access.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Convert to allowlist: `match normalized_membership_role(...) { Some("translator" \| "member" \| "admin" \| "owner") => Ok(()), _ => Err(...) }` | Consistent with every other gate in the file; new roles denied by default |
| B | Add `"unknown"` explicitly to the denylist | Closes immediate gap but is still denylist-based; needs updates for future unrecognized roles |
| C | Add `"unknown"` denylist + improved error message directing user to refresh team access | Marginally better diagnostics; same structural weakness as B |

**Recommended**: A

### M2 — `broker_auth.rs:60–72`, `github/app_auth.rs:35–40`

**No timeout on any reqwest blocking client; indefinite UI hang on unresponsive broker**

The `reqwest::blocking::Client` has no default timeout. Every Tauri command that calls the broker (inspect session, refresh session, every installation access check) blocks the Tauri command thread until the server responds or the OS TCP stack gives up — potentially minutes on a hung or unreachable server.

The two inline client builders in `broker_auth.rs` also duplicate the `github_client()` pattern already established in `github/app_auth.rs`.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Add `.timeout(Duration::from_secs(30))` to a shared `broker_client() -> Result<Client, String>` factory in `broker.rs`; replace inline builders in `broker_auth.rs` with calls to this factory | Timeout policy defined once; eliminates DRY violation |
| B | Add `.timeout(Duration::from_secs(30))` inline at each call site | Closes the hang; leaves three scattered policy points |

**Recommended**: A

### M5 — `installation_access.rs:253–269`

**ensure_snapshot_allows_team_ai_access skips snapshot_has_verified_membership; undocumented structural asymmetry**

Every other `ensure_snapshot_allows_*` function calls `snapshot_has_verified_membership` first and returns `UNVERIFIED_ACCESS_ERROR` before checking the specific permission. The team-AI checker inspects `membership_role` directly, skipping this shared verification path. | Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Route through `snapshot_has_verified_membership` first, then check viewer: `if !snapshot_has_verified_membership(snapshot) { return Err(UNVERIFIED_TEAM_ACCESS_ERROR); } if is_read_only_membership_role(...) { return Err(READ_ONLY_TEAM_AI_ERROR); } Ok(())` | Uniform trust model; capability-only owners treated consistently |
| B | Add a prominent comment explaining why capability-flag owners are excluded from team-AI (if intentional) plus an explicit test | Preserves current behaviour if intentional; prevents future "fix" |

**Recommended**: A if the exclusion is a bug; B if it is intentional policy that has never been documented.

### M6 — `installation_access.rs:36, 43–49, 145–160`

**cached_at field is written but never read; refresh always does a live broker round-trip**

`cached_at` is written at line 60 but is never read anywhere in the codebase. `refresh_installation_access_snapshot` unconditionally makes a blocking network call on every invocation — there is no TTL check and no cache reuse. The field name and function name `cache_installation_access` imply a caching mechanism that does not exist, misleading future developers.

Every write-gated operation (commit, glossary write, QA write, etc.) incurs a synchronous, blocking broker round-trip before proceeding.

| Fix | Description | Rationale |

| **A ✓** | Implement the TTL: in `refresh_installation_access_snapshot`, read the on-disk snapshot first; if `cached_at` is recent (e.g. < 60 s), return it without a network call. Add a forced-refresh variant for explicit "refresh access" user actions. | Eliminates one blocked network round-trip per write operation; honours the API the field names already advertise. Combine with M2's timeout fix for bounded latency. |
| B | Remove `cached_at` from the struct; rename `cache_installation_access` → `store_installation_access` | Honest naming; defer caching to a future implementation |

**Recommended**: A for new implementation; B immediately if A is deferred.

### M7 — `ai_secret_storage.rs:139–145`

**load_or_create_client silently swallows errors; corrupt snapshot appears as vanished secrets**

```rust
stronghold.get_client(AI_SECRET_CLIENT_ID)
    .or_else(|_| stronghold.load_client(AI_SECRET_CLIENT_ID))
    .or_else(|_| stronghold.create_client(AI_SECRET_CLIENT_ID))
```

If the snapshot file exists but is corrupt, or if `stronghold_password` produces a different hash (e.g. if the snapshot path changes between app versions), both `get_client` and `load_client` fail with errors that are silently discarded. `create_client` then creates an empty store, making every `load_store_value` return `None`. From the user's perspective, all AI secrets have vanished. The actual root cause is permanently masked.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Distinguish "client not found" from "client exists but failed to load": if `load_client` returns a "not found" error, fall through to `create_client`; if it returns any other error, surface it to the caller | Corruption is surfaced as an error; only genuine first-use falls through to create |
| B | Log intermediate errors (`tracing::warn!`) before falling through | Adds diagnostic trail without restructuring; still masks error from caller |

**Recommended**: A (B acceptable as interim if the Stronghold error API does not expose a distinguishable "not found" variant)

### m3 — `broker_auth_storage.rs:40`, `installation_access.rs:290`

**Non-atomic file writes; partial write on process crash leaves unparseable files**

`fs::write` truncates then writes. If the process is killed between truncation and completion, the resulting file is empty or partial. The next `load_broker_auth_session` fails to parse it (returns an error, not `None`). The same pattern exists in `installation_access.rs:290`.

| Fix | Description | Rationale |
|---|---|---|
| **A ✓** | Write to `.tmp` sibling then `fs::rename` atomically: `let tmp = path.with_extension("json.tmp"); fs::write(&tmp, contents)?; fs::rename(&tmp, &path)?;` | `fs::rename` is atomic on POSIX; caller always sees old or new file, never partial |
| B | Treat parse failure on load as `Ok(None)` and auto-delete the corrupt file | Lower effort; tolerates rare corruption but loses the data |

**Recommended**: A
