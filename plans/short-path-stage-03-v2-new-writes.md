# Short Path Stage 3: New V2 Writes

## Summary

Make newly created or newly imported repos write the v2 layout from the start. This stage changes new writes only; legacy repos are still handled by the migration stages before release.

## Scope

- New project, glossary, and QA repos write `.gtms/repo.json`.
- New local checkout folders use short unique names.
- New project chapter folders use short unique chapter slugs.
- New project images are stored flat in the chapter image folder.
- Full human-readable names stay in metadata and display JSON.

## Implementation Details

Existing code touchpoints:

- Project repo initialization lives in `src-tauri/src/project_import/chapter_editor/mod.rs` at `initialize_gtms_project_repo_sync`.
- Glossary repo initialization lives in `src-tauri/src/glossary_storage/mod.rs` at `initialize_gtms_glossary_repo_sync` and TMX import initialization.
- QA list repo initialization lives in `src-tauri/src/qa_list_storage/mod.rs` at `initialize_gtms_qa_list_repo_sync` and TMX import initialization.
- Chapter import writes are in `src-tauri/src/project_import/chapter_import/write_gtms.rs`.
- Editor image upload writes are in `src-tauri/src/project_import/chapter_editor/images.rs`.
- Frontend repo names are generated by `src-ui/app/repo-names.js`; those names remain GitHub repo names and should not be changed to the 22-character local folder rule.

New repo initialization:

- Project, glossary, and QA create/import flows write `.gtms/repo.json` immediately.
- New metadata should include:
  - `schemaVersion: 1`
  - correct `repoKind`
  - `storageLayoutVersion: 2`
  - `appliedMigrations: ["0.8.10"]`
- Initial commit for a new repo should include the metadata file.
- Stage `.gtms/repo.json` alongside `.gitattributes` and the root resource JSON.
- Keep initialization commit messages the same unless release notes need a visible migration marker; the critical machine-readable values are `.gtms/repo.json` and the existing `GTMS-App-Version` trailer.

Local checkout folders:

- Allocate local folders with the shared short folder helper.
- Collision checks are local and case-insensitive.
- Keep GitHub repo names unchanged.
- Store the resolved local folder in local sync state.
- `prepare_local_gtms_glossary_repo_sync` and the QA equivalent currently allocate `repo_root.join(repo_name)` before initialization; they need to call the shared local-folder allocator.
- Project initialization currently falls back to `repo_root.join(repo_name)`; change that fallback to the allocated short local folder.

Project chapter imports:

- Allocate chapter folder names with the shared short folder helper.
- Store the full title in `chapter.json`.
- Store the actual folder slug in `chapter.json` only where the current schema requires it.
- Avoid using the title-derived folder as a stable identity.
- Replace `unique_chapter_slug` in `chapter_import/write_gtms.rs` with the shared short-folder allocator.
- Keep `chapter_id` as the durable chapter identity and keep `find_chapter_path_by_id` as the lookup mechanism.
- Do not rename a chapter folder on ordinary chapter title rename; existing rename only updates `chapter.json`, and that should remain true.

Editor and importer image writes:

- Store uploaded/imported images at:

```text
chapters/<chapter-folder>/images/<image-file-name>
```

- Allocate image names with the shared image filename helper.
- Allocate image names inside the repo write queue so concurrent uploads cannot choose the same filename.
- Update row `image.path` to the new repo-relative path.
- Use the resolved chapter folder path, not `chapterId`, when building filesystem paths.
- Replace `relative_uploaded_image_path` in `chapter_editor/images.rs`; it currently builds `chapters/{chapter_id}/images/row-{row_id}-{language}-{uuid}/{file}` even though the chapter folder may not equal `chapter_id`.
- Replace `relative_imported_image_path` in `chapter_import/write_gtms.rs`; it currently creates nested `images/row-*` directories.
- The allocator should read existing files from `chapter_path.join("images")` immediately before writing so it sees files created by previous queued operations.
- Image remove/replace code must continue removing the old path stored in the row, including old-layout paths during the transition before Stage 4/5.

Temporary compatibility note:

- During this stage, old-layout reads may still exist to keep development incremental.
- The release must not ship until Stage 4 and Stage 5 ensure old-layout repos are migrated or blocked before normal use.

## Existing-Code Risks

- Frontend `slugifyRepoName` deliberately keeps GitHub repo names up to 100 characters. Do not reuse the 22-character local-folder helper for GitHub repo names or this change would unexpectedly rename remote repos.
- The editor upload path currently uses `chapter_id`; that only works accidentally for some old paths. After short slugs, any code that joins `chapters/<chapterId>` will fail unless it uses `find_chapter_path_by_id`.
- `chapter_export.rs` builds raw GitHub image URLs from stored row image paths. It should continue to work automatically if row paths are updated, but tests should cover flat v2 paths.
- `row_uploaded_image_relative_paths` and image removal code must support both old and new paths until migration is enforced everywhere.
- `ensure_gitattributes` currently tracks `assets/** binary`, while uploaded images live under `chapters/**/images/**`. If binary handling matters on Windows/Git, consider whether `.gitattributes` should include image paths in this stage.
- If image filename allocation happens outside the repo write queue, two quick uploads with the same original name can choose the same flat filename.

## Tests

- New project repo includes `.gtms/repo.json`.
- New glossary repo includes `.gtms/repo.json`.
- New QA repo includes `.gtms/repo.json`.
- New chapter import creates a short folder while preserving the full title in `chapter.json`.
- New image upload writes to the flat chapter image folder.
- Duplicate image names in the same chapter resolve with `-2`, `-3`, etc.
- Concurrent image name allocation is serialized through the repo write queue.
- Chapter rename does not rename the short chapter folder.
- Exported HTML/raw GitHub image URLs work for flat v2 image paths.
- Replacing/removing an image cleans up the old uploaded file path without removing unrelated files in the flat `images` folder.

## Acceptance Criteria

- All new repos and new project content use v2 layout.
- Human-readable names continue displaying untruncated in the UI.
- GitHub repo names remain unchanged.
